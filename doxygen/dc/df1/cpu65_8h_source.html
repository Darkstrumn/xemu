34 /* Notes:
35  * cpu flags (pf_*) are INT in type, and "logic" types not actual bit mask.
36  this is because using flags are frequent and faster (maybe ... ?) this way,
37  and actual getting of the full flags register is much rare event when we
38  need to "construct" it from individual flag info data.
39  Type is "int" since in some cases they're implemented with > 8 bit data
40  length steps during opcode emulation.
41  * if CPU65_DISCRETE_PF_NZ is *NOT* defined:
42  pf_nz is an exception to the scheme above: 65xx CPU family is famous for
43  altering N and Z flags always if a general purpose register is filled,
44  even with eg TXA. So this is the way more frequent opcode flag operation
45  in general. Thus, I included two bits at the right bit position for N and
46  Z but not for the other flags though (see my previous comment).
47  * OK, maybe my theory is not the best, and it's not even optimal this way,
48  I have to admit ...
49  * 'B' bit (break flag) is not a real processor status flag. it depends on
50  the type of operation what the result is, eg BRK pushes it into the stack as
51  '1', while hardware interrupts as '0'
52 */
